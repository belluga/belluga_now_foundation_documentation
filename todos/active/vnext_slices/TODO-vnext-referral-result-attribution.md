## Objective
Define a future-proof, MongoDB-friendly attribution model that allows Belluga to answer questions like:

- "How many 1st-level check-ins did this inviter generate?"
- "How many 1st-level promo requests came from this invite tree?"
- "How many 2nd-level check-ins did this inviter indirectly generate?"

without requiring runtime graph traversal or recursive query logic on hot paths.

This is a compatibility/design stream only. It is **not** an MVP implementation requirement.

---

## 1. Problem Statement

Belluga already models invites as a directed referral graph. That graph is enough to answer direct invite acceptance attribution, but future product and business flows will also want attributed downstream "results" generated by that invite tree, such as:

- check-ins,
- promo requests,
- purchases,
- offer claims,
- future micro-conversions.

If those questions are answered by walking the invite graph at request time, MongoDB query cost will grow quickly and create pressure for `$graphLookup`, fan-out joins, or repeated recounts. That is not acceptable for hot dashboards, rankings, or event/operator analytics.

The design must stay:

- simple,
- append-only where possible,
- package-agnostic,
- efficient for MongoDB.

---

## 2. Foundational Direction

- [x] âś… `RRA-01` Canonical state remains package-owned.
  - Invites owns invite acceptance attribution.
  - Future check-in/check-out/promo/purchase packages own their own canonical state.
  - A shared activity fact layer must never become the source-of-truth for domain status.

- [x] âś… `RRA-02` Attribution beyond direct invite acceptance is derived from append-only activity facts.
  - Downstream "results" are modeled as activity facts, not as ad hoc counters scattered across packages.

- [x] âś… `RRA-03` Referral lineage is snapshotted, not recomputed at read time.
  - The system should persist bounded lineage information when a credited invite is accepted, then reuse that lineage for downstream result attribution.

- [x] âś… `RRA-04` MongoDB hot reads must use indexed facts/projections, not graph traversal.
  - No request-path `$graphLookup`
  - No recursive tree expansion on dashboards or event analytics
  - No runtime recount over raw invite edges for hot metrics

- [x] âś… `RRA-05` Supported lineage depth is intentionally bounded.
  - Belluga should support level-based attribution (`level 1`, `level 2`, optionally `level 3`) through a configured cap instead of unbounded ancestry.

## 3. Minimal Conceptual Model

- [ ] âšŞ `credited_acceptance`
  - a canonical accepted invite edge that owns referral credit for a `(receiver_user_id, target_ref)`

- [ ] âšŞ `referral_lineage_snapshot`
  - a bounded ancestor list stored when a credited acceptance is established
  - example:
    - receiver `B` accepted from inviter `A` -> lineage = `[A:l1]`
    - receiver `C` accepted from inviter `B` -> lineage = `[B:l1, A:l2]`

- [ ] âšŞ `activity_fact`
  - append-only record describing a downstream result produced by an actor in an event/occurrence context
  - examples:
    - `check_in.recorded`
    - `promo.requested`
    - `purchase.completed`
    - `offer.claimed`

- [ ] âšŞ `principal_conversion_metrics`
  - aggregated projection keyed by principal + target + result type + level
  - example:
    - inviter `A`
    - `event_id = X`
    - `result_type = check_in.recorded`
    - `level = 2`
    - `count = 30`

## 4. Canonical Flow

### 4.1 On credited invite acceptance

- [ ] âšŞ When a credited acceptance is recorded, persist a bounded referral lineage snapshot on the accepted invite edge or on a tightly coupled acceptance document.
- [ ] âšŞ The lineage must prepend the current credited inviter as `level = 1`, then shift prior ancestry by `+1`.
- [ ] âšŞ The lineage must be truncated to the configured max supported depth.

Example:

- `A -> B` accepted
  - lineage for `B` = `[A:l1]`
- `B -> C` accepted
  - lineage for `C` = `[B:l1, A:l2]`
- `C -> D` accepted with max depth = 2
  - lineage for `D` = `[C:l1, B:l2]`

### 4.2 On downstream result generation

- [ ] âšŞ When an actor generates a downstream result inside the event/occurrence context, emit/store an `activity_fact`.
- [ ] âšŞ The fact must snapshot the actor's current referral lineage for that same target context.
- [ ] âšŞ The fact must be idempotent and append-only.

Example:

- `C` performs `check_in.recorded`
  - activity fact includes:
    - actor = `C`
    - result type = `check_in.recorded`
    - lineage = `[B:l1, A:l2]`

### 4.3 Projection update

- [ ] âšŞ Projection workers consume activity facts and increment/update `principal_conversion_metrics`.
- [ ] âšŞ For every lineage entry in the activity fact, update the corresponding `(principal, result_type, level, target_ref)` bucket.

So for `C` check-in with lineage `[B:l1, A:l2]`:

- increment `B / check_in.recorded / level 1`
- increment `A / check_in.recorded / level 2`

## 6. MongoDB Query & Projection Strategy

- [x] âś… `RRA-06` Activity facts are append-only and queryable, but not the only read surface.
  - Use facts for drill-down, audits, and projection input.
  - Use aggregated projections for dashboards, rankings, and summary counters.

- [x] âś… `RRA-07` Hot reads must hit precomputed projections.
  - Example questions:
    - "How many level-1 check-ins did inviter A generate for event X?"
    - "How many level-2 promo requests did inviter A generate this month?"
  - These should resolve from `principal_conversion_metrics`, not from raw facts and not from invite graph recursion.

- [x] âś… `RRA-08` Raw facts support filtered drill-down by lineage token.
  - Example:
    - query `lineage_tokens = "user:A:l2"` for level-2 attributed facts

- [x] âś… `RRA-09` The system must not materialize a global transitive closure table.
  - Only bounded lineage snapshots per accepted actor/target and per activity fact are allowed.

- [ ] âšŞ Do not use `$graphLookup` for invite-result attribution on request paths.

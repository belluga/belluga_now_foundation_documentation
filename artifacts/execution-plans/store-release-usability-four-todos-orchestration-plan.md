## Authority Boundary

- Governing TODOs define **WHAT** must be delivered and what counts as done.
- This plan defines **HOW** the orchestrator will dispatch workers, reconcile worktrees, validate runtime behavior, and record evidence for the four current Store Release usability TODOs.
- This plan supersedes `store-release-usability-orchestration-plan.md`, `store-release-event-programming-navigation-orchestration-plan.md`, and `store-release-event-rich-text-public-rendering-orchestration-plan.md` as execution authority for this four-TODO wave. Those artifacts may remain historical context only.
- If this plan conflicts with a governing TODO, execution stops until the TODO or this plan is updated. Requirement wording in the governing TODOs is literal.
- Replacing a named artifact, UI control, navigation path, runtime target, validation lane, or browser evidence requirement requires an approved row in the Spec Deviation Ledger.
- Implementation belongs to workers/subagents. The orchestrator owns dispatch, worker coordination, reconciliation, merge-conflict resolution when unavoidable, final runtime validation, guard execution, checkpoint hygiene, and evidence consolidation.
- Each visible item requires Playwright navigation and/or ADB integration evidence according to the TODO matrix. Code lines, unit tests, widget tests, analyzer, or serialized payload evidence are supporting evidence only and cannot close visible browser/device behavior.
- Workers must add fail-first tests for their owned rows before implementation where practical, and every row must have either a concrete runtime test or a recorded reason why the row is structure-only.
- CRUD or mutation-visible behavior requires real local mutation evidence on a non-main lane; public/admin visible rendering requires final Web build evidence with `bash scripts/build_web.sh ../web-app dev` before Playwright.

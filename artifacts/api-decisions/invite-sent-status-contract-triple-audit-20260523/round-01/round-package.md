# Invite Sent Status Contract Decision Package - 2026-05-23

## Decision Scope
Evaluate whether the current `GET /invites/sent-statuses` contract should remain the target design or be replaced/refined before promotion.

This is a strategic/API-contract decision package only. No code change is authorized by this artifact.

## Current Baseline
- Current implemented endpoint: `GET /invites/sent-statuses`.
- Current purpose: hydrate canonical sent invite status for the authenticated inviter, scoped by `occurrence_id`, with optional `event_id` consistency context and optional `recipient_account_profile_ids[]` filter.
- Current max result rule: unfiltered response returns up to 200 items; internally one extra item can be fetched to detect truncation.
- Current client usage:
  - Invite composer calls status hydration with visible `recipient_account_profile_ids[]`.
  - Event detail can call occurrence-level status hydration without a recipient filter to render summary/footer.
  - `invite_accepted` push refreshes only the affected occurrence and optionally one recipient.
- Current module authority:
  - `invite_and_social_loop_module.md` defines `/contacts/inviteables` as the canonical composer source for in-app recipients.
  - The same module defines `/invites/sent-statuses` as occurrence-scoped sent invite state.

## Problem Statement
The current endpoint is good enough to fix local-only sent invite state, but it may mix two different read models:

1. **Composer actionability:** for a paginated/visible list of inviteable recipients, each row needs exact `sent_invite_status` / `blocks_reinvite`.
2. **Event summary widget:** event detail needs counters plus a small avatar preview, not up to 200 full sent-invite rows.

The question is whether the target design should:
- keep the current dedicated hydration endpoint;
- move status enrichment into the paginated `/contacts/inviteables` query;
- split summary/preview from row-level actionability.

## Options

### Option A - Keep Current Dedicated Hydration Endpoint
Keep `GET /invites/sent-statuses` as the primary read contract for both composer actionability and event summary/footer.

Expected shape:
- `occurrence_id` required.
- optional `recipient_account_profile_ids[]`.
- returns `summary` computed from returned rows plus up to 200 `items`.

Benefits:
- Minimal code churn.
- Already implemented and tested.
- Direct occurrence-scoped lookup avoids page-walking.
- Works well for push refresh and visible-recipient hydration.

Risks:
- Unfiltered summary can be truncated and not represent the full occurrence when more than 200 sent invites exist.
- Event detail/footer overfetches when it needs only counters and preview.
- Composer still performs two conceptual reads: inviteables first, statuses second.
- The endpoint can become a pseudo-list without real pagination.

### Option B - Move Row Status Into Paginated `/contacts/inviteables`
Make `/contacts/inviteables` accept `event_id`/`occurrence_id` context and return each paginated inviteable row with `sent_invite_status`, `blocks_reinvite`, `sent_at`, `responded_at`, and related status metadata.

Expected shape:
- `/contacts/inviteables?occurrence_id=...&cursor=...`
- items are paginated inviteable recipients already enriched with sent invite state for the authenticated inviter.
- summary is absent or included only as a separate aggregate object.

Benefits:
- Best fit for composer: status travels with the row being rendered.
- Avoids fetching status for users not on the current page.
- Preserves pagination as the volume-control boundary.
- Reduces client orchestration complexity for invite buttons.

Risks:
- Expands `/contacts/inviteables` from relationship discovery into occurrence-specific actionability.
- Needs careful aggregation/indexing across inviteable recipient sources and `invite_edges`.
- Can make a hot endpoint heavier for every composer page.
- Push/event-detail summary still needs a separate read model.

### Option C - Split Read Models
Keep separate contracts for separate jobs:

1. Composer row actionability:
   - `/contacts/inviteables` gains optional occurrence context and returns paginated rows enriched with `sent_invite_status`.
2. Summary/preview:
   - new or refactored occurrence summary contract returns exact counters over the whole occurrence plus limited preview, e.g. first 5 avatars.
3. Dedicated status hydration:
   - `GET /invites/sent-statuses` remains narrow only for targeted refresh by `recipient_account_profile_ids[]`, push reconciliation, and compatibility during rollout; it is not used as the event summary source and does not pretend to be a paginated list.

Benefits:
- Best semantic separation.
- Exact counters can be computed via aggregation over the whole occurrence without row overfetch.
- Composer remains paginated and row-owned.
- Push refresh remains cheap and targeted.
- Avoids turning any one endpoint into a catch-all read model.

Risks:
- More contracts to implement and test.
- Requires frontend routing of each surface to the correct read model.
- Needs migration from current event-detail use to summary endpoint.

## Evaluation Criteria
- Correctness: no stale local-only status; no false invite button enablement.
- Performance: no unbounded scans, no page-walking, no N+1 projection, no 200-row overfetch for summary-only UI.
- API clarity: each endpoint should have one primary read-model responsibility.
- Frontend simplicity: row actionability should be available when rendering inviteable rows.
- Push compatibility: accepted push should refresh only the affected occurrence/recipient.
- Promotion risk: avoid destabilizing the already-green fix unless the current contract is a real blocker.

## Preliminary Delphi Assessment
- Option A is acceptable as a tactical bugfix and is already green, especially when called with `recipient_account_profile_ids[]`.
- Option A is not the best target design for event summary/footer because summary can be bounded to returned rows and the endpoint can overfetch.
- Option B is strong for composer actionability but incomplete for event summary and push refresh.
- Option C is the best target architecture if we want clean long-term semantics:
  - paginated inviteables own row actionability;
  - occurrence summary owns counters/preview;
  - sent-status hydration remains targeted, not a broad list or summary source.
